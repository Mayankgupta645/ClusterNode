How Attackers Move Through Your Cluster

1
Initial Compromise
A vulnerability gives an attacker code execution inside a Pod. Traditional tools see an isolated container incident. ClusterNodes sees the first node of an attack graph.
CVE Exploit · RCE
2
ServiceAccount Token Theft
The attacker reads the auto-mounted token from /var/run/secrets. One over-privileged ServiceAccount binding becomes the bridge to the entire cluster.
Token Exfiltration
3
RBAC Privilege Escalation
The token carries a ClusterRole with secrets:get and pods:exec. The attacker can now enumerate every Secret cluster-wide.
Privilege Escalation

Why is the api-server pod flagged critical? What's the actual risk?
The api-server Pod is bound to svc-admin with a ClusterRoleBinding to cluster-admin. This creates a 4-hop critical path:

Pod → svc-admin → cluster-admin → prod-db-creds → postgres

An attacker can extract your production database password in under 60 seconds. Apply this fix:
# Restrict ServiceAccount role binding apiVersion: rbac.authorization.k8s.io/v1 kind: RoleBinding metadata: name: api-server-restricted namespace: production roleRef: kind: Role name: api-read-only
Apply the fix and show remaining blast radius.
